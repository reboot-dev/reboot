# UIs for AI Chat Apps

UI methods declare a React component that opens in the AI chat
interface when the AI calls the associated tool. Unlike `reader`,
`writer`, `transaction`, and `workflow` methods, UI methods have
no backend servicer code — the React `App.tsx` component **is**
the implementation.

:::info MCP only
UI methods are only callable over MCP — they appear as tools in
AI chat interfaces. There is no way to invoke a `UI` method
from a browser or backend.
:::

### Parameters

| Parameter | Type | Description |
| --- | --- | --- |
| `request` | `Model` subclass or `None` | If provided, the AI passes parameters to the React component as props. If `None`, the UI takes no parameters from the AI. |
| `path` | `str` | Path to the React app directory relative to the project root. Must contain an `index.html`. |
| `title` | `str` or `None` | Human-readable title shown in tool listings. |
| `description` | `str` or `None` | Description used by the AI to decide when to open this UI. Write it as instructions to the AI. |

## React implementation

Each `UI` method points to a directory containing:
- `index.html` — the Vite entry point.
- `main.tsx` — renders the React component wrapped in
  `RebootClientProvider`.
- `App.tsx` — the component that implements the UI.

The component uses generated hooks (like `useCounter()`) to
interact with backend state. The state ID is automatically
provided by the framework when the UI is opened — you do
not need to pass it.

```tsx
import { useCounter } from "@api/ai_chat_counter/v1/counter_rbt_react";

export const ClickerApp: FC = () => {
  const counter = useCounter();
  const { response } = counter.useGet();

  const handleIncrement = async () => {
    await counter.increment({ amount: 1 });
  };

  return (
    <div>
      <span>{response?.value ?? 0}</span>
      <button onClick={handleIncrement}>+</button>
    </div>
  );
};
```

## How it works under the hood

When you define a `UI` method, Reboot generates two MCP
artifacts:

1. **An MCP tool** that the AI calls to open the UI. The tool
   returns metadata telling the AI client to display the visual
   interface.
2. **An MCP resource** that serves the compiled HTML/JS/CSS. The
   AI client loads this resource in an embedded browser.

The React app communicates with your Reboot backend via
WebSocket subscriptions (for reads) and the appropriate write
channel (for mutations).

## Bundling UI methods

<!-- TODO(rjh): replace this whole section with an `npm create` that scaffolds this stuff -->

Each `UI` method's React app must be bundled into a single HTML
file so that the Reboot server can serve it as an MCP resource.
Use [Vite](https://vite.dev/) with
[`vite-plugin-singlefile`](https://www.npmjs.com/package/vite-plugin-singlefile).

### Dependencies

```sh
npm install -D @vitejs/plugin-react vite vite-plugin-singlefile
```

### Vite configuration

Create a `vite.config.ts`. It auto-discovers your UIs from the `mcp/` directory (each
subdirectory with an `index.html`):

```ts title=frontend/vite.config.ts
// Vite configuration for Reboot UIs.
//
// One config drives two jobs, selected by `command`/`mode`:
//
//   * `vite` (serve): a single dev server that delivers HMR for every
//     `mcp/<name>` UI under `base: "/__/frontend/"`. Envoy proxies that
//     prefix to this dev server (`run --config=hmr`).
//   * `vite build --mode <name>`: builds one MCP UI into a single,
//     self-contained `dist/mcp/<name>/index.html` (assets inlined via
//     `vite-plugin-singlefile`). The framework serves it at
//     `/__/frontend/mcp/<name>/index.html` in dist mode.
import fs from "fs";
import path from "path";
import react from "@vitejs/plugin-react";
import { defineConfig } from "vite";
import { viteSingleFile } from "vite-plugin-singlefile";

// Auto-discover MCP UIs: every `mcp/<name>/` with an `index.html`.
// There may be no `mcp/` directory at all (one whose last MCP UI was
// removed), so guard the read.
const mcpDir = path.resolve(__dirname, "mcp");
const mcpNames: string[] = fs.existsSync(mcpDir)
  ? fs
      .readdirSync(mcpDir)
      .filter((name) => fs.existsSync(path.resolve(mcpDir, name, "index.html")))
  : [];

// Path alias for API imports (`@api/...` -> `./api/...`).
const resolve = {
  alias: {
    "@api": path.resolve(__dirname, "./api"),
  },
  dedupe: ["react", "react-dom", "zod"],
};

export default defineConfig(({ command, mode }) => {
  // Dev server: serves the MCP UIs with HMR.
  //
  // UIs use a double iframe architecture:
  //   MCP Host -> srcdoc (origin=null) -> iframe (origin=localhost:9991)
  //
  // The inner iframe loads from Envoy ("/__/frontend/**"), which
  // proxies to Vite. Because the inner iframe has a real origin,
  // Vite's URLs work normally. `base: "/__/frontend/"` ensures all
  // paths route through Envoy.
  //
  // Hot Module Replacement works automatically: Vite's client connects
  // to the page's origin, and Envoy proxies WebSocket upgrades to
  // Vite. This also works with tunnels (ngrok) since the tunnel
  // points to Envoy.
  if (command === "serve") {
    const port = parseInt(process.env.RBT_VITE_PORT || "4444", 10);

    return {
      plugins: [react()],
      root: ".",
      resolve,
      base: "/__/frontend/",
      server: {
        port,
        strictPort: true,
        // Listen on all interfaces since requests come through
        // Envoy (and tunnels).
        host: true,
        allowedHosts: true,
      },
    };
  }

  // Build one MCP UI: `vite build --mode <name>`. We root the build at
  // `mcp/<name>/` so the output lands directly at
  // `dist/mcp/<name>/index.html`, a single self-contained file (assets
  // inlined by `vite-plugin-singlefile`). The framework serves it at
  // `/__/frontend/mcp/<name>/index.html` in dist mode.
  const name = mode;
  if (!mcpNames.includes(name)) {
    const valid = mcpNames.map((n) => `--mode ${n}`).join(", ");
    throw new Error(
      `Unknown MCP UI: ${mode || "(unset)"}. Use one of: ${valid}.`
    );
  }

  return {
    plugins: [react(), viteSingleFile()],
    root: path.resolve(__dirname, "mcp", name),
    base: "/__/frontend/",
    build: {
      outDir: path.resolve(__dirname, "dist/mcp", name),
      emptyOutDir: true,
      assetsInlineLimit: 100000000,
      cssCodeSplit: false,
      rollupOptions: {
        output: {
          inlineDynamicImports: true,
        },
      },
    },
    resolve,
  };
});
```

### Building

Build one UI by name with `--mode mcp:<name>` (or build every UI and
the `web/` SPA at once with `npm run build`):

```sh
cd frontend && npx vite build --mode mcp:clicker
```

The bundled HTML file is written to `frontend/dist/mcp/clicker/index.html`.
The Reboot server discovers and serves it as an MCP resource.

## No backend servicer

`UI` methods are skipped in your servicer class. You only
implement `reader`, `writer`, `transaction`, and `workflow`
methods in the servicer.
