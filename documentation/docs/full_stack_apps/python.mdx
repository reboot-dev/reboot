<Step
  title="Define your durable state data types and operations"
  description={`

First, import Reboot's \`Model\`, from which you will subclass your
requests and responses.

Next, import the necessary types to define your API from \`reboot.api\`.

Next, define your durable state type in Reboot called \`ChatRoomState\` by
creating a \`ChatRoomState\` class subclassed from \`Model\`. This is where
your \`messages\` will be stored.

Next, define the methods on your data type calling the \`Methods\` function
and passing to it the name of your methods, what type of method it is:
\`Reader\`, \`Writer\`, \`Transaction\` or \`Workflow\`, a \`request\`
and a \`response\` classes.

Lastly, define your API by passing the API \`Type\`, \`ChatRoom\`, to the \`API\`
function.

:::info Naming conventions
Your state class should end in \`State\`, e.g., \`ChatRoomState\`. Your \`Methods\`
variable should end in \`Methods\`, e.g., \`ChatRoomMethods\`. Your \`API\` types
are composed of \`state\` and \`methods\` and should be descriptive of the
interface, e.g., \`ChatRoom\`. Following these naming conventions will make your
backend code easy to understand.
:::

For brevity we are defining just two operations:

1. \`messages\` returns all of your messages. It is a
[\`Reader\`](/learn_more/define/methods#kinds) because it only reads state.
1. \`send\` takes a \`message\` and updates state, so it is a
[\`Writer\`](/learn_more/define/methods#kinds) method.

In addition to the operations, we need to define any supporting
\`classes\`'s, e.g., \`SendRequest\`, \`SendResponse\`, etc.

:::tip
[\`Reader\`](/learn_more/define/methods#kinds) methods safely execute concurrently!
:::

`}>

```python title=./api/chat_room/v1/chat_room.py
# ./api/chat_room/v1/chat_room.py
from reboot.api import (
    API,
    Field,
    Methods,
    Reader,
    Model,
    Writer,
    Type,
)
from typing import Optional

class ChatRoomState(Model):
    messages: Optional[list[str]] = Field(tag=1)

class MessagesResponse(Model):
    messages: list[str] = Field(tag=1)

class SendRequest(Model):
    message: str = Field(tag=1)

ChatRoomMethods = Methods(
    messages=Reader(
        request=None,
        response=MessagesResponse,
    ),
    send=Writer(
        request=SendRequest,
        response=None,
    ),
)

api = API(
    ChatRoom=Type(
        state=ChatRoomState,
        methods=ChatRoomMethods,
    ),
)
```

<Step
  title={<div>Implement <code>ChatServicer</code></div>}
  description={`

To start, your servicer class must extend \`ChatRoom.Servicer\`.

An instance of your servicer will be created for every instance of the
\`ChatRoom\` type that you create, and you can access the state of that
instance via \`self.state\`.

Each method expects a \`context\` as the first argument that aligns
with the function given in the \`.py\` file; \`Messages\`
takes a \`ReaderContext\` and \`Send\` takes a \`WriterContext\`.

\`Send\` can mutate \`self.state\` and \`Messages\` can only read from
it.

:::note
When \`request\` is \`None\`, no request argument is passed to the method.
The method should return \`None\` if \`response\` is \`None\`, otherwise
an instance of the \`response\` type.
:::

`}>

```python title=./backend/src/chat_room_servicer.py
# ./backend/src/chat_room_servicer.py
from chat_room.v1.chat_room import (
    MessagesResponse,
    SendRequest,
)
from chat_room.v1.chat_room_rbt import ChatRoom
from reboot.aio.contexts import ReaderContext, WriterContext


class ChatRoomServicer(ChatRoom.Servicer):

    async def messages(
        self,
        context: ReaderContext,
    ) -> MessagesResponse:
        return MessagesResponse(messages=self.state.messages or [])

    async def send(
        self,
        context: WriterContext,
        request: SendRequest,
    ) -> None:
        if not self.state.messages:
            self.state.messages = []

        self.state.messages.append(request.message)
```

</Step>

<Step
  title={<div>Implement <code>main.py</code></div>}
  description={`

\`Application\` takes a list of the Python
[\`servicers\`](/learn_more/implement/servicers) that you have implemented.

It optionally takes an \`initialize\` function that runs before the app loads,
which can be used to construct default state.

`}>

```python title=./backend/src/main.py
# ./backend/src/main.py
import asyncio
from chat_room.v1.chat_room_rbt import ChatRoom
from chat_room_servicer import ChatRoomServicer
from reboot.aio.applications import Application
from reboot.aio.external import InitializeContext


async def initialize(context: InitializeContext):
    chat_room = ChatRoom.ref('reboot-chat-room')

    await chat_room.send(
        context,
        message='Hello, World!',
    )


async def main():
    application = Application(
        servicers=[ChatRoomServicer],
        initialize=initialize,
    )

    await application.run()


if __name__ == '__main__':
    asyncio.run(main())
```

</Step>

<Step
  title={<div>Add necessary flags to your <code>.rbtrc</code> file</div>}
  description={`

First, tell Reboot where the proto files live, the \`api/\` directory,
and where to output the generated code, \`backend/api/\`.

Next, tell Reboot how to run your app. Give the app a name via
\`--application-name=chat-room\` to persist state between app restarts. Watch
for source code changes via \`--watch=...\` to restart the app whenever there
is a modification. Finally, tell Reboot that it should use Python to run
our app.

:::tip
If you want to remove all of your changes between app restarts, do not pass the
\`--application-name\` flag to \`rbt\`.
:::

Last, specify the entrypoint for your app.

See a complete list of supported flags by running \`rbt dev run --help\`.

`}>

```sh
generate api/
generate --python=backend/api/

dev run --application-name=chat-room
dev run --watch=backend/src/**/*.py
dev run --python
dev run --application=backend/src/main.py
```

</Step>
